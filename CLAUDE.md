# Petit Crazy Arcade 작업 규칙

Godot 4.7 / GDScript / GdUnit4 6.1.3. **TDD로 진행합니다.**

셋업, 실행, 디렉터리 구조는 `docs/development.md`에 있습니다. 여기에는 규칙만 둡니다.
매 세션 로드되는 문서이므로, 지키지 않으면 사고가 나는 것만 적습니다.

## 작업 흐름

**한 사이클은 작게 잡습니다.** "물풍선을 놓는다"까지가 한 사이클이고 "물풍선이 터진다"는 다음
사이클입니다. 한 번에 규칙 두 개를 넣으면 빨강이 왜 빨강인지 흐려집니다.

1. 빨강 테스트를 먼저 씁니다.
2. **정말 빨강인지, 그리고 맞는 이유로 빨강인지 눈으로 확인합니다.** "함수가 없다"는 에러도
   빨강이지만, 단언이 일을 시작하는 건 구현이 생긴 다음입니다.
3. 최소한으로 초록을 만듭니다.
4. 이름과 중복을 정리합니다. 그동안 테스트는 계속 초록이어야 합니다.
5. 커밋합니다.

## 아키텍처 원칙 (테스트 가능성이 여기서 결정됨)

- `src/core/`는 게임 규칙입니다. `RefCounted` 기반이고 **엔진에 의존하지 않습니다**
  (Node, 씬 트리, `_process` 없음). `Foo.new()`로 만들어 즉시 단언할 수 있어야 합니다.
  로직의 대부분이 여기 있어야 정상입니다.
- `src/game/`은 Node 레이어입니다. 입력을 받고 `src/core/`를 호출하고 화면에 반영하는
  **얇은 껍데기**입니다.
- 도메인 로직은 시그널 대신 **값을 반환**하세요 (`func tick(delta) -> Array[WaterBalloon]`).
  반환값은 바로 단언되지만 시그널은 구독하고 기다려야 합니다.
- **게임 상태는 씬 루트 Node가 소유합니다** (`Main`이 `Character`를 듦). `RefCounted`는 참조가
  끊기면 사라지므로, 프레임을 넘어 살아남으려면 Node가 붙들어야 합니다. 의존 방향은 한쪽입니다.
  Node에서 core로는 되고, core에서 Node로는 안 됩니다.
- **화면 노드(뷰)는 상태를 소유하지 않습니다.** `CharacterView`는 시키는 자리에 그리기만 합니다.
  진실은 `Character`(격자 좌표)에 있고, 뷰는 그것을 픽셀로 비춘 그림자입니다.
  `*_view.position`을 직접 만지는 코드는 "core 좌표를 픽셀로 반영하는" 한 곳 말고는 없어야 합니다.

## 용어

도메인은 한국어, 코드는 영어입니다. **새 개념의 이름은 `docs/development.md`의 용어표에서
가져오고, 표에 없으면 표부터 갱신하세요.** 즉석에서 지으면 같은 것을 두 낱말로 부르게 됩니다.
표에는 이미 쓰이는 이름과, 아직 코드에 없지만 예약해 둔 이름(`WaterStream`, `Bubble`)이 있습니다.

## 하네스 관련 함정 (실제로 겪은 것들)

- **`class_name`을 새로 추가하면 `godot --headless --path . --import`를 한 번 돌려야 합니다.**
  안 하면 `Identifier "Foo" not declared` 라는 엉뚱한 파싱 에러가 납니다.
- 씬 테스트에서 `runner.scene()`을 **그 스크립트의 타입으로 받으려면 `class_name`이 필요합니다.**
  없으면 `Node2D`로밖에 못 받고, 스크립트가 추가한 멤버는 자동완성도 타입 검사도 안 됩니다.
  런타임엔 동작해서 더 헷갈립니다.
- `bin/test.sh`에는 `-c`(fail-fast 해제)가 들어 있습니다. GdUnit4는 기본이 fail-fast라서
  첫 실패 시 스위트의 나머지 테스트를 실행하지 않고, 리포트에는 *실행된* 수만 찍혀
  테스트가 사라진 것처럼 보입니다.
- `--remote-debug tcp://127.0.0.1:0`도 필수입니다. 없으면 파싱 에러가 났을 때 Godot이 대화형
  디버거로 진입하면서 **종료 코드를 0으로 뭉갭니다.** CI가 깨진 빌드에 초록불을 줍니다.
  그래서 테스트 로그 맨 위의 이 두 줄은 **정상입니다. 고치려 들지 마세요.**
  포트 0은 바인딩되지 않고, 그 **연결 거부가 곧 디버거를 못 붙게 하는 장치**입니다.
  ```
  ERROR: The remote port number must be between 1 and 65535 (inclusive).
  ERROR: Remote Debugger: Unable to connect to host '127.0.0.1:0'.
  ```
- headless에서는 `InputEvent`가 전달되지 않습니다. 입력 시뮬레이션 테스트는 `bin/test.sh --gui`로.
  애초에 입력에 직접 의존하는 테스트를 만들지 않는 게 낫습니다.
- **엔진 의존은 껍데기로 밀어내세요.** `_unhandled_input(event)`는 "키 눌림인지 보고 키코드를
  꺼내 넘긴다"만 하고, 실제 처리는 `Key`를 인자로 받는 함수에 둡니다. 그 함수는 `InputEvent` 없이
  직접 호출할 수 있어 **headless 씬 테스트에서 검증됩니다.** `_process(delta)`가 `tick(delta)`를
  부르는 것도 같은 모양입니다. 씬 테스트는 `_main.tick(4.0)`을 직접 불러 프레임을 기다리지 않고
  결정론적으로 단언합니다. core와 game의 경계에서 한 번, Node 레이어 안에서 또 한 번,
  같은 원칙의 반복입니다.
- 대신 대가는 알고 치릅니다. **엔진 콜백 그 한 줄은 어떤 테스트도 지켜주지 않습니다.**
  `_process`가 통째로 사라져도 테스트는 전부 초록이고 게임만 멈춥니다.
  그래서 그 껍데기는 **반드시 한 줄로 유지하세요.** 한 줄에는 버그가 숨을 데가 없습니다.
- 시간이나 `delta`에 기댄 결과를 씬 테스트에서 단언하면 flaky합니다.
  시간 규칙은 `src/core/`에서 결정론적으로 테스트하세요.
- `simulate_frames(5)` 후 프레임 수를 **정확히 5**로 단언하지 마세요. `--gui`로 창을 띄우면
  실제 엔진 루프도 프레임을 돌려서 더 커집니다. headless에서만 통과하는 테스트가 됩니다.
  `is_greater_equal(5)`처럼 하한만 단언하세요.

## 테스트 작성 규칙 (초록불을 의심하는 법)

- **기준은 하나입니다. 그럴듯한 틀린 구현이 이 테스트에서 빨강이 되는가.**
  안 된다면 그 테스트는 자기 이름이 약속한 것을 지키지 못합니다.
- **실패할 수 없는 단언을 쓰지 마세요.** "같은 칸에 또 놓으면 무시한다"를
  `has_water_balloon(cell) == true`로 단언하면 무시하든 덮어쓰든 두 개 쌓이든 전부 초록입니다.
  규칙을 관측 가능한 값으로 바꿔 물어야 합니다. `water_balloon_count() == 1`처럼요.
- **테스트 이름이 코드보다 많이 약속하지 않게 하세요.** `..._맵_밖으로_이동하지_못한다`가
  초록이었지만 실제로 막던 것은 위와 왼쪽 두 변뿐이었습니다(`max(ZERO)`). 오른쪽과 아래는
  무방비였고 테스트는 그것을 몰랐습니다.
- **좌표 단언에 대칭 값을 쓰지 마세요.** `Vector2i.ONE`(1, 1)은 x와 y가 같아서 좌표를
  전치(transpose)한 구현도 통과시킵니다. `Vector2i(3, 5)`처럼 비대칭 값으로 쓰세요.
- **시간은 실제 경로대로 흘리세요.** `Main`에서 `Map`으로, `Map`에서 `WaterBalloon`으로
  흐릅니다. 테스트가 중간을 건너뛰고 `water_balloon.tick()`을 직접 부르면 **맵이 시간을
  전파하는지를 아무도 검증하지 않습니다.** 테스트는 초록인데 게임에서는 아무도 tick해주지 않아
  물풍선이 영원히 안 터집니다.
- **시간 단언은 경계를 정확히 때리지 마세요.** `tick(3.0)`이 `POP_AFTER_SECONDS`와 같으면
  `>=`냐 `>`냐라는 구현 세부에 테스트가 매달립니다. 확실히 모자란 시간에는 안 터지고 확실히
  넘는 시간에는 터진다는 식으로 여유를 두세요. 단언은 언제나 **불리언**(터졌다/안 터졌다)에
  겁니다. 남은 시간을 float으로 비교하면 드리프트에 물립니다.
- **누적을 검증하려면 각각은 문턱보다 작고 합계는 문턱을 넘어야 합니다.** 안 그러면 "delta를
  그냥 상수와 비교"하는(누적하지 않는) 구현이 통과합니다. 실제 게임에서 `delta`는 0.016초씩
  들어오므로 그 구현으로는 영원히 안 터집니다.

## 좌표계 (격자와 픽셀을 섞지 마세요)

- **격자 좌표(칸 번호)는 `Vector2i`(정수), 화면 좌표는 `Vector2`(픽셀)** 입니다.
- 둘 사이의 변환은 **`Map.to_pixel()` 한 곳에서만** 합니다. 이 함수의 시그니처
  (`Vector2i`를 받아 `Vector2`를 돌려주는)가 두 세계의 국경 검문소입니다.
- **타입이 안 맞아 생기는 마찰을 "통일"로 없애려 하지 마세요.** 그 마찰이 방어막입니다.
  실제로 한 번 전부 `Vector2`로 바꿨다가 되돌렸습니다. 입력과 출력 타입이 같아지면
  **이미 픽셀인 값을 `to_pixel()`에 다시 넣어도 컴파일러가 안 막습니다.**
  그때 **테스트는 20개 전부 초록불이었습니다.** 초록불이 옳다는 뜻은 아닙니다.
- 칸 번호를 float으로 두면 나중에 부드러운 이동(`속도 × delta` 누적)을 붙일 때 드리프트가 쌓여
  "정확히 3번 칸인가"를 물을 수 없게 되고, 칸을 딕셔너리 키로 쓸 수 없습니다.
  (물풍선을 어느 칸에 놓을지 판정할 때 터집니다.)
- 시간과 거리의 누적 일반에 대해서도 같습니다. 부동소수점 드리프트를 항상 의심하세요.
