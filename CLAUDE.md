# Petit Crazy Arcade — 작업 규칙

Godot 4.7 / GDScript / GdUnit4 6.1.3. **TDD로 진행합니다.**

## 아키텍처 원칙 (테스트 가능성이 여기서 결정됨)

- `src/core/` — 게임 규칙. `RefCounted` 기반. **엔진 의존 금지** (Node, 씬 트리, `_process` 없음).
  `Foo.new()`로 만들어 즉시 단언할 수 있어야 합니다. 로직의 대부분이 여기 있어야 정상입니다.
- `src/game/` — Node 레이어. 입력을 받고 `src/core/`를 호출하고 화면에 반영하는 **얇은 껍데기**.
- 도메인 로직은 시그널 대신 **값을 반환**하세요 (`func tick(delta) -> Array[Explosion]`).
  반환값은 바로 단언되지만 시그널은 구독하고 기다려야 합니다.
- **게임 상태는 씬 루트 Node가 소유합니다** (`Main`이 `Character`를 듦). `RefCounted`는 참조가
  끊기면 사라지므로, 프레임을 넘어 살아남으려면 Node가 붙들어야 합니다. 의존 방향은 한쪽입니다 —
  Node → core는 되고, core → Node는 안 됩니다.
- **화면 노드(뷰)는 상태를 소유하지 않습니다.** `CharacterView`는 시키는 자리에 그리기만 합니다.
  진실은 `Character`(격자 좌표)에 있고, 뷰는 그것을 픽셀로 비춘 그림자입니다.
  `*_view.position`을 직접 만지는 코드는 "core 좌표 → 픽셀 반영" 한 곳 말고는 없어야 합니다.

## 하네스 관련 함정 (실제로 겪은 것들)

- **`class_name`을 새로 추가하면 `godot --headless --path . --import`를 한 번 돌려야 합니다.**
  안 하면 `Identifier "Foo" not declared` 파싱 에러가 납니다.
- 씬 테스트에서 `runner.scene()`을 **그 스크립트의 타입으로 받으려면 `class_name`이 필요합니다.**
  없으면 `Node2D`로밖에 못 받고, 스크립트가 추가한 멤버는 자동완성도 타입 검사도 안 됩니다
  (런타임엔 동작해서 더 헷갈립니다).
- `bin/test.sh`에는 `-c`(fail-fast 해제)가 들어 있습니다. GdUnit4는 기본이 fail-fast라서
  첫 실패 시 스위트의 나머지 테스트를 실행하지 않고, 리포트에는 *실행된* 수만 찍혀
  테스트가 사라진 것처럼 보입니다.
- `--remote-debug tcp://127.0.0.1:0`도 필수입니다. 없으면 파싱 에러 시 Godot이 대화형
  디버거로 진입하면서 **종료 코드를 0으로 뭉갭니다** → CI가 깨진 빌드에 초록불을 줍니다.
  그래서 테스트 로그 맨 위의 이 두 줄은 **정상입니다. 고치려 들지 마세요.**
  포트 0은 바인딩되지 않고, 그 **연결 거부가 곧 디버거를 못 붙게 하는 장치**입니다.
  ```
  ERROR: The remote port number must be between 1 and 65535 (inclusive).
  ERROR: Remote Debugger: Unable to connect to host '127.0.0.1:0'.
  ```
- headless에서는 `InputEvent`가 전달되지 않습니다. 입력 시뮬레이션 테스트는 `bin/test.sh --gui`로.
  애초에 입력에 직접 의존하는 테스트를 만들지 않는 게 낫습니다.
  → **엔진 의존을 껍데기로 밀어내세요.** `_unhandled_input(event)`는 "키 눌림인지 보고 키코드를
  꺼내 넘긴다"만 하고, 실제 처리는 `Key`를 인자로 받는 함수에 둡니다. 그 함수는 `InputEvent` 없이
  직접 호출할 수 있어 **headless 씬 테스트에서 검증됩니다.** core/game 경계에서 한 번,
  Node 레이어 안에서 또 한 번 — 같은 원칙의 반복입니다.
- 시간/`delta` 기반 결과를 씬 테스트에서 단언하면 flaky합니다.
  시간 규칙은 `src/core/`에서 결정론적으로 테스트하세요.
- `simulate_frames(5)` 후 프레임 수를 **정확히 5**로 단언하지 마세요. `--gui` 로 창을 띄우면
  실제 엔진 루프도 프레임을 돌려서 더 커집니다. headless에서만 통과하는 테스트가 됩니다.
  `is_greater_equal(5)` 처럼 하한만 단언하세요.

## 좌표계 — 격자와 픽셀을 섞지 마세요

- **격자 좌표(칸 번호)는 `Vector2i`(정수), 화면 좌표는 `Vector2`(픽셀)** 입니다.
- 둘 사이의 변환은 **`Map.to_pixel()` 한 곳에서만** 합니다. 이 함수의 시그니처
  (`Vector2i` → `Vector2`)가 두 세계의 국경 검문소입니다.
- **타입이 안 맞아 생기는 마찰을 "통일"로 없애려 하지 마세요.** 그 마찰이 방어막입니다.
  실제로 한 번 전부 `Vector2`로 바꿨다가 되돌렸습니다. 입력과 출력 타입이 같아지면
  **이미 픽셀인 값을 `to_pixel()`에 다시 넣어도 컴파일러가 안 막습니다.**
  그때 **테스트는 20개 전부 초록불이었습니다** — 초록불이 옳다는 뜻은 아닙니다.
- 칸 번호를 float으로 두면 나중에 부드러운 이동(`속도 × delta` 누적)을 붙일 때
  드리프트가 쌓여 "정확히 3번 칸인가"를 물을 수 없게 되고, 칸을 딕셔너리 키로 쓸 수 없습니다.
  (물풍선을 어느 칸에 놓을지 판정할 때 터집니다.)
- 시간·거리 누적 일반에 대해서도 같습니다. 부동소수점 드리프트를 항상 의심하세요.
